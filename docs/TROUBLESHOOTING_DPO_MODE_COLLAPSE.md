# Troubleshooting: DPO 모드 붕괴 (Mode Collapse)

---

## 1. 문제 발견

파인튜닝 파이프라인의 마지막 단계인 DPO(Direct Preference Optimization) 학습을 2 에포크로 실행한 후, `compare.py`로 생성 결과를 확인했을 때 다음과 같은 출력이 관찰됐다.

```
[AFTER] (((((((((((((((((((((((((((((((((((((((((((((((((((((((((
```

모든 프롬프트에 대해 AFTER 모델이 의미 없는 반복 문자만 생성했다. BEFORE 모델(사전학습 체크포인트)은 주제와 무관하지만 최소한 자연어를 생성한 반면, AFTER 모델은 완전히 붕괴된 상태였다.

---

## 2. 문제 확인

학습 로그를 역추적하여 DPO 에포크별 손실 값을 확인했다.

| 에포크 | 평균 DPO Loss |
|--------|--------------|
| 1      | 0.6031       |
| 2      | 0.3471       |

에포크 2의 개별 스텝 손실을 보면 `0.054`, `0.083` 등 극단적으로 낮은 값이 반복됐다. 정상적인 DPO 학습에서 loss가 이 수준으로 내려가면 모델이 reference policy로부터 너무 멀리 벗어났다는 신호다.

붕괴한 모델 파일(`finetuned_dpo2ep.json`, 2.8GB)은 백업으로 보존했다.

---

## 3. 가설 수립 및 해결 방안 검토

### 가설: DPO loss 과소수렴으로 인한 분포 붕괴

DPO의 목적함수는 다음과 같다.

```
L_DPO = -log σ(β · (log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x)))
```

`β`(KL 페널티 계수)가 작을수록, 선호 데이터 쌍이 적을수록, 에포크를 반복할수록 모델이 reference policy로부터 급격히 이탈할 수 있다. loss가 0에 가까워진다는 것은 모델이 선호/비선호 쌍을 너무 강하게 분리했다는 의미이며, 이 과정에서 token probability가 특정 토큰(여기서는 `(`)에 극단적으로 몰리는 붕괴가 발생했다.

### 검토한 해결 방안

| 방안 | 설명 | 단점 |
|------|------|------|
| A. `--dpo-epochs 1`로 재실행 | 안전한 loss 구간(0.6031)에서 중단 | 근본적 해결 아님 |
| B. `β` 값 상향 조정 | KL 페널티 강화로 이탈 억제 | 하이퍼파라미터 재탐색 필요 |
| C. 선호 데이터 대폭 증가 | 500-5,000쌍으로 DPO 재실험 | `build_dpo.py` 작성 + 장시간 학습 필요 |
| D. SFT만으로 마무리 | DPO 단계 생략 | 선호 정렬 효과 없음 |

---

## 4. 해결 방안 선택의 근거

**방안 A(`--dpo-epochs 1`)를 선택**했다. 근거는 다음과 같다.

- 현재 DPO 데이터셋은 **58쌍**으로, curriculum 권장량(5,000–20,000쌍)의 1% 수준이다. 데이터가 이 수준에서는 어떤 하이퍼파라미터 조정도 임시방편이다.
- 에포크 1의 loss(0.6031)는 학습이 진행됐지만 붕괴 직전이라는 신호가 없는 안전한 구간이었다.
- 프로젝트 목표는 "파이프라인 완성 및 전후 비교"이지 최적 DPO 성능이 아니다. 현 단계에서 추가 실험에 시간을 쓰는 것보다 Project ①을 완성하는 것이 우선이었다.
- 방안 C는 중기 과제로 별도 계획(`build_dpo.py`로 500쌍 생성)이 이미 로드맵에 포함되어 있다.

---

## 5. 해결 과정

```bash
# 붕괴 모델 백업
mv checkpoints/finetuned.json checkpoints/finetuned_dpo2ep.json

# DPO 1 에포크로 재실행
uv run python finetune.py \
    --ckpt checkpoints/ckpt_step200000.pt \
    --tok  tokenizer.model \
    --data data/sft_data.jsonl \
    --dpo-data data/dpo_data.jsonl \
    --dpo-epochs 1 \
    --out  checkpoints/finetuned.json
```

**재실행 결과:**

| 단계 | 결과 |
|------|------|
| SFT epoch 3 avg loss | 2.0308 (ppl 7.62) |
| DPO epoch 1 avg loss | 0.6031 ← 안전 구간 |
| LoRA 병합 | 완료 (추론 오버헤드 없음) |
| 저장 파일 | `checkpoints/finetuned.json` (2.8GB) |

`compare.py` 결과, AFTER 모델이 `SFT`, `LoRA`, `rank=16`, `loss scaling`, `DPO` 등 파인튜닝 도메인 어휘를 생성하는 것을 확인했다. 완전한 의미 있는 문장 생성에는 이르지 못했지만, 모드 붕괴는 해소됐다.

## 6. 결론 및 다른 가설

### 결론

DPO 모드 붕괴의 직접 원인은 **데이터 부족(58쌍) 상태에서 에포크 과다(2회)**였다. 에포크를 1회로 제한하여 붕괴를 방지하고 파이프라인을 완성했다.

### 남은 가설 및 후속 실험 아이디어

1. **β 탐색**: 현재 β 기본값이 낮게 설정되어 있을 수 있다. β를 0.1 → 0.5 → 1.0으로 높이면서 58쌍에서도 안정적인 2에포크 학습이 가능한지 실험할 수 있다.

2. **데이터 스케일**: `build_dpo.py`로 Claude API를 활용해 500쌍 생성 후 재실험. 데이터가 충분해지면 에포크 제한 없이 학습 가능한지 확인.

3. **Early stopping**: DPO loss가 특정 임계값(예: 0.3) 이하로 내려가면 자동으로 학습을 중단하는 로직 추가. 에포크 수 대신 loss를 기준으로 중단점 결정.

4. **Reference model 고정 확인**: LoRA 학습 중 reference policy가 실제로 고정되어 있는지 파라미터 그래디언트 추적으로 검증 필요.
