# Troubleshooting: LLM-as-judge 평가 전패 (BEFORE/AFTER 모두 1점)

---

## 1. 문제 발견

`eval.py`로 LLM-as-judge(Claude Haiku) 평가를 실행한 결과, BEFORE/AFTER 모델 모두 5개 질문에서 전부 1점을 받았다.

```
BEFORE 평균: 1.00점
AFTER  평균: 1.00점  (Δ +0.00)
Win/Tie/Loss: 0/5/0  (Win rate 0%)
```

---

## 2. 문제 확인

모델 출력 예시:

| 질문 | BEFORE 출력 | AFTER 출력 |
|------|------------|-----------|
| SFT가 뭐야? | "의 모든 부분이 다 들어맞는다. 이 중 하나는 2개의 단서이며..." | "인(Hering)은 모델이 학습 중 모델에 호출하고..." |
| LoRA가 뭐야? | "microsoft Digital Design은 일반적인 리눅스 커널에..." | "LoRA는 훈련 중 선택할 수 있는 모든 에이전트(예: Chatte-level)를..." |
| RAG란? | "NASA는 모든 사람들에게, 모두 알고 있다!" | "R G는(Hy 최적화)와(Residit)가 RG의 첫 번째 단계인..." |

AFTER 모델은 AI/ML 어휘(LoRA, DPO, 에이전트 등)가 일부 등장하지만 의미 있는 문장을 생성하지 못한다.

---

## 3. 가설 수립 및 원인 분석

### 가설 A: SFT 포맷 불일치

`finetune.py`가 학습한 포맷과 `eval.py`가 사용하는 추론 포맷이 다를 수 있다.
- 학습 시: `{"user": "질문", "assistant": "답변"}` 형식으로 연결하여 학습
- 추론 시: 모델이 동일한 포맷의 프롬프트를 받아야 답변 부분을 생성하는데, 포맷이 맞지 않으면 엉뚱한 토큰을 생성

### 가설 B: 모델 규모 한계

현재 모델은 GPT-2 수준(수천만~수억 파라미터)으로, scratch 사전학습 단계의 데이터와 학습량이 부족하다.
- 기반 모델 자체가 자연어 생성 능력을 충분히 습득하지 못한 상태
- SFT는 이미 언어를 잘 생성하는 모델의 "방향"을 조정하는 기법이며, 기반 능력이 없으면 효과 없음

### 가설 C: SFT 데이터 부족

552쌍은 instruction-following 능력을 학습하기에 턱없이 부족하다.
- 상용 모델(LLaMA, Mistral 등)의 SFT는 수만~수십만 쌍 사용
- 현재 552쌍으로는 어휘 분포 이동(vocabulary shift)은 가능하지만 Q&A 패턴 학습은 어려움

---

## 4. 해결 방안 검토

| 방안 | 설명 | 예상 효과 | 비용 |
|------|------|---------|------|
| A. 추론 포맷 수정 | `eval.py`/`compare.py` 프롬프트를 학습 포맷과 일치 | 즉각적, 저위험 | 낮음 |
| B. SFT 데이터 대폭 증가 | 552 → 5,000쌍 (`build_sft_v2.py` 재실행) | 중간 | API 비용 |
| C. 사전학습 강화 | 더 많은 corpus로 base 모델 재학습 | 근본 해결 | 매우 높음 |
| D. HuggingFace 기반 모델 사용 | GPT-2/LLaMA 공개 체크포인트에 SFT 적용 | 즉각적, 고효과 | 낮음 |

---

## 5. 우선 조치

**방안 A 먼저 확인** — 가장 빠르고 비용 없음.

`finetune.py`의 학습 포맷과 `eval.py`의 추론 포맷을 비교하여 불일치 여부 확인.

불일치가 없다면 → **방안 B** (데이터 5,000쌍으로 증량) 또는 **방안 D** (HuggingFace base 모델 교체) 검토.

---

## 6. 결론 및 후속 실험 아이디어

### 결론

현재 결과는 GPT-2 수준 scratch 모델 + 552 SFT쌍의 구조적 한계에서 비롯된다. 파이프라인 자체(SFT → DPO → 평가)는 정상 작동이 확인됐으며, 성능 향상을 위해선 데이터 스케일 또는 기반 모델 교체가 필요하다.

### 후속 실험 아이디어

1. **포맷 검증**: `finetune.py` chat template 확인 → `eval.py` 프롬프트와 동일하게 맞추기
2. **데이터 스케일**: `build_sft_v2.py --rag-files 200 --rag-pairs 10`으로 2,000쌍 생성 후 재학습
3. **HuggingFace 기반 모델**: 현재 numpy 엔진 대신 `transformers` + `GPT2LMHeadModel` + LoRA(peft)로 파이프라인 재구성
4. **손실 마스킹 검증**: SFT 학습 시 user 턴에 loss mask가 올바르게 적용되는지 확인
